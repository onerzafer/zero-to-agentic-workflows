🧩 Exercise 1: Session Memory (In-Memory Arrays)

Goal: Maintain conversation context within the active thread by appending messages to a local Javascript array buffer.

Implementation Details

The developer implements a function handleSessionMemory(messages, userMessage, assistantMessage). If the assistant's reply is not yet known, we push the user query to build the prompt payload. Once the model responds, we append the response to close the loop.

function handleSessionMemory(messages, userMessage, assistantMessage = null) { if (assistantMessage === null) { messages.push({ role: 'user', content: userMessage }); return messages; } else { messages.push({ role: 'assistant', content: assistantMessage }); return messages; } }

💾 Exercise 2: Persistent Log (Naïve File I/O)

Goal: Erase session boundary limits by saving the conversation transcript to disk (memories.md) and feeding it back into the system prompt on new launches.

Implementation Details

Before querying the API, the app reads memories.md and injects its contents as a history reference in the system instructions. When the reply is returned, the new user-assistant dialogue pair is appended to the file on disk.

function handlePersistentLog(userMessage, assistantMessage = null) { if (assistantMessage === null) { let memories = ""; if (fs.existsSync(MEMORY_FILE_PATH)) { memories = fs.readFileSync(MEMORY_FILE_PATH, 'utf8'); } return `You are a helpful assistant. You have access to past log memories:\n${memories}`; } else { const logEntry = `\n--- Chat Log ---\nUser: ${userMessage}\nAssistant: ${assistantMessage}\n`; fs.appendFileSync(MEMORY_FILE_PATH, logEntry, 'utf8'); } }

⚙️ Exercise 3: Tool-Calling Memory (Agentic Interception Loop)

Goal: Optimize token costs and context precision. Instead of dumping raw logs, the LLM uses local tools (save_memory and search_memory) to write and search memories on demand.

Implementation Details

This creates a recursive **Interception Loop**. When the model returns a tool_calls array, the client executes the tool locally (reading/writing to memories.md), appends the result to history with role tool, and re-queries the model. The loop repeats until the LLM returns a text response.

async function handleToolCallingMemory(userMessage) { toolHistory.push({ role: 'user', content: userMessage }); const payload = { model: selectedModel, messages: [ { role: 'system', content: instructions }, ...toolHistory ], tools: MEMORY_TOOLS }; let data = await makeOpenRouterCall(payload); let message = data.choices[0].message; while (message.tool_calls && message.tool_calls.length > 0) { toolHistory.push(message); // Save tool calls request for (const toolCall of message.tool_calls) { const name = toolCall.function.name; const args = JSON.parse(toolCall.function.arguments); const result = executeLocalTool(name, args); // Run local fs code toolHistory.push({ role: 'tool', tool_call_id: toolCall.id, name: name, content: result }); } // Re-query LLM with tool responses data = await makeOpenRouterCall({ ...payload, messages: [systemMsg, ...toolHistory] }); message = data.choices[0].message; } toolHistory.push(message); // Save final text response return message.content; }

Exercise 2 Discussion

Topic: Log-All Persistence (Naïve Memory)

If we record every chat turn to a file and read it all back as background context, what happens to our token usage and bill as the chat grows? What if the user says "I hate apples" and then later "I love apples"—how does the LLM resolve conflicting history?

Exercise 3 Discussion

Topic: Tool-Calling Memory (Selective Memory)

If the AI uses tools to search for "ice cream" in the file, but the text was written as "likes gelato", will text-search find it? How does semantic search (vector embeddings) solve this? How do we prevent the AI from calling tools on trivial statements like "I am tired"?
